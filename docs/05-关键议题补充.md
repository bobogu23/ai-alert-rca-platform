## 1. 兜底设计、Human-in-the-Loop 与运行时安全

### 1.1 分层兜底体系

本项目遵循 "**任何单点失败都不崩溃、不阻塞、不静默失败**" 的原则, 兜底分层落在四个层级 (对应详细设计文档第 7 章):

```
┌─────────────────────────────────────────────────────────────┐
│ 层 4  全局兜底: FallbackReportBuilder (纯本地渲染, 不依赖 LLM)   │← 整体超时/推理失败/三路证据全空
├─────────────────────────────────────────────────────────────┤
│ 层 3  流程降级: Critic 未通过 -> 置信度封顶 LOW/MEDIUM         │← Critic 循环超轮次/超时/不可用
├─────────────────────────────────────────────────────────────┤
│ 层 2  Agent 隔离: 单 Agent 失败 -> 该维度证据缺失, 其余继续     │← Future.get(timeout) 允许部分失败
├─────────────────────────────────────────────────────────────┤
│ 层 1  调用容错: RetryExecutor + TimeLimiter (MCP/LLM)         │← 网络抖动/5xx/限流/瞬时故障
└─────────────────────────────────────────────────────────────┘
```

**关键设计点** (均对应真实代码):

- **层 1 — RetryExecutor + TimeLimiter**: `RetryableException` 才重试, 指数退避 200/400/800ms + jitter, 不可重试异常直接抛出避免放大故障;
- **层 2 — Orchestrator.await**: 每路 Agent 独立虚拟线程提交, `future.get(agentMs)` 超时即 `future.cancel(true)` 释放资源, 转为 `Optional.empty()`, `EvidenceBundle.hasAnyEvidence()` 判断是否走全局兜底;
- **层 3 — criticLoop.capConfidence**: Critic 循环超轮次/超时时, 保留当前推理结果但把所有 `HIGH` 置信度**强制封顶为 `LOW`/`MEDIUM`**, 不给出"看似高置信实则未经校验"的结论;
- **层 4 — FallbackReportBuilder**: 纯本地渲染 markdown, 显式标注 "系统未能完成自动归因, 请人工排查", 置信度 LOW, `fallback=true`, 案卷置为 `ANALYZE_FAILED`. **绝不静默丢弃**, 让告警不会因归因失败而消失.

### 1.2 Human-in-the-Loop 设计

本项目定位是**"资深 SRE 的智能副驾"** (架构文档 §1.3), 不做自愈变更、不替代最终决策. HITL 体现在三个环节:

#### 1.2.1 报告层 — 显式的人工介入信号

报告结构自带"是否需要人工"的强信号, 让 SRE 在企微群一眼分辨:

| 信号 | 触发条件 | SRE 应做 |
|---|---|---|
| `fallback=true` | 整体超时/推理失败/证据全无 | 立即人工介入, 系统未产出结论 |
| `confidence=LOW` + Critic 未通过标记 | Critic 循环打回超轮次 | 参考证据自行判断, 不采信结论 |
| "迭代取证未证实明确方向" 降级说明 | `InvestigationOutcome.confirmedCause` 为空 | 结论仅供参考, 需人工进一步取证 |
| "Critic 校验不可用, 置信度不高于 MEDIUM" | Critic Agent 异常 | 结论未经反幻觉校验, 手动核对证据链 |

所有降级理由都通过 `RcaReport.degradeNote` 拼接后展示, **透明可见, 绝不隐藏**.

#### 1.2.2 反馈回路 — SRE 反哺系统

在 `NotificationGateway` 推送的企微 markdown 尾部提供三个交互按钮:

- **采纳** → 写入 `alert_case.feedback = ADOPTED`, 计入采纳率;
- **驳回** → 弹出原因单选 (根因错/证据不对/建议不可行/其他), 写入 `feedback_reason`, 该样本自动进入回归集候选;
- **需人工深入** → 触发案卷标记 `NEEDS_MANUAL`, 供事后复盘统计.

#### 1.2.3 边界: 不做"分析前审批", 但做"推送前分级"

第一版明确**不引入报告推送前的人工审批环节** (架构文档 §1.3), 原因是**及时性优先**——排障场景下多等 5 分钟人工审批, 归因价值就归零. 替代方案是:
- 推送前按置信度**分级投放**: `HIGH` 直发主群; `MEDIUM/LOW` 或兜底报告发**值班专属通道**, 避免噪音打扰全组;
- 高危场景 (如生产环境 + 建议中含"回滚/降配") 可通过配置开关升级为"值班人先看再转发", 但此为可选项而非默认.

### 1.3 超时问题的解决

**核心原则: 超时优先, 宁可降级也不无限等待.**

分层预算 (`RcaProperties.Timeout`, 满足 `mcp < llm < agent`, `overall` 覆盖多轮 Critic):

| 层 | 预算 | 实现位置 | 超时行为 |
|---|---|---|---|
| MCP 单次调用 | 10s | `TimeLimiter.callWithTimeout` | 抛 `TimeLimitExceededException`, 可重试 |
| LLM 单次调用 | 60s | `LlmClient.callWithRetryAndTimeout` | **不重试**, 直接抛 `RcaException` 交上层降级 |
| Agent 单次执行 | 70s | `Orchestrator.await` | `future.cancel(true)` + 该维度证据置空 |
| Critic 循环 | 70s (含 ≤ 2 轮) | `criticLoop.deadline` | 保留当前结果 + 置信度封顶 LOW |
| 整体分析 | 180s | `runSafely.timeLimiter` | 走 `FallbackReportBuilder` 全局兜底 |
| HTTP 读超时 | 65s (略大于 llm) | `spring.http.client.read-timeout` | 底层兜底, 防 `TimeLimiter` 外仍有阻塞 |

**LLM 超时刻意不重试的理由**: 超时意味着模型或网关此刻确实慢, 重试大概率再次超时并把整体耗时**放大 2~3 倍**, 反而让"该走兜底"的场景走不到; 故 `LlmClient` 捕获 `TimeLimitExceededException` 时**转为不可重试异常**, 交上层降级. 这是"故障期间不放大对下游压力"原则的具体落地.

**双层重试避免**: Spring AI 内置重试通过 `spring.ai.retry.max-attempts: 1` 显式关闭, 只保留应用侧 `RetryExecutor`, 防止"框架 × 应用"叠加导致故障期间调用次数被数倍放大.

### 1.4 死循环 / 无界迭代的防护

系统中所有"看似可能循环"的位置都设了**硬上限 + 死线**双保险:

| 循环点 | 硬上限 | 死线 | 收敛策略 |
|---|---|---|---|
| **Critic 循环** | `critic.max-iterations = 2` | `criticLoopMs = 70s` deadline | 超限即保留当前结果 + 置信度封顶 |
| **迭代取证** | `investigation.max-rounds = 3` | 每轮 MCP 调用带 `mcpCallMs` 超时 | 超限用当前证据收敛; 决策树用 `Set<SuspectedCause> visited` 防重复方向 |
| **RetryExecutor** | `llm.max-retries = 2` | 无 (由 TimeLimiter 卡外层) | 超限抛出 `RetryableException` 上抛 |
| **提交队列** | `queueCapacity = 128` | 无 (由 overall 超时卡内层) | 超限直接 `RejectedExecutionException` → 兜底 |

关键: **每一处循环, 若既能到达硬上限也能到达死线, 都会被强制打断**——`for (round = 1; round <= maxRounds; round++)` 与 `System.currentTimeMillis() > deadline` 双条件, 缺一不可.

**规则决策树防打转**: `InvestigationService.nextTarget` 用 `visited` 集合确保每个方向至多验证一次, 遍历完当前 chain 无未验证候选则返回 `UNKNOWN` 结束, 从算法层面**根除**打转可能.

### 1.5 其它运行时安全议题

- **背压保护**: `submissionExecutor` 有界并发 + 有界队列, 队列满即拒, 避免告警风暴压垮下游 (详细设计 §7.9);
- **幂等**: `RcaCacheService` 按 `(caseId, agentName)` 缓存中间结果, TTL 30 分钟, 同案卷重跑不重复付费 LLM;
- **异常吞噬红线**: 所有 `catch (RuntimeException)` **必带日志 + 计入指标**, 不允许静默吞;
- **中断响应**: 所有虚拟线程任务在 `InterruptedException` 时 `Thread.currentThread().interrupt()` 恢复中断标志, 保证 `future.cancel(true)` 能真正终止阻塞的 IO.

## 2. 大模型选型

### 2.1 选型的四个约束

本项目对模型的诉求同时受四个约束制约, 任何单点最优都无意义:

1. **推理质量**: 需要跨证据 (日志/指标/代码/告警) 的**因果推理**能力, 而非简单摘要;
2. **结构化输出稳定性**: 所有 Agent 输出都要通过 Spring AI `BeanOutputConverter` 解析成 Java Record, JSON Schema 不能漂移;
3. **延迟可控**: 整体 180s 预算内需要跑一次路由 + 三路 Agent + 一次推理 + 最多 2 轮 Critic = **6~8 次 LLM 调用**, 单次不能太慢;
4. **成本可承受**: 生产峰值日均可能 500~1000 例归因, 每例 6~8 次调用, 强模型单价必须能承受.

### 2.2 分档路由策略 (ModelTier)

本项目已实现**双档路由**, 由 `ModelTier` + `LlmClient.resolveModel` 承载:

| 档位 | 用途 (Agent) | 温度 | 配置项 | 选型倾向 |
|---|---|---|---|---|
| `REASONING` (强模型) | `RootCauseReasoningAgent`, `CriticAgent` 语义校验 | 0.2 (低温, 稳输出) | `rca.llm.reasoning-model` | 优先推理能力, 次考虑成本 |
| `LIGHTWEIGHT` (轻量) | `RoutingEngine` LLM 兜底、日志/Trace/代码 Agent 内部的分类研判 (`suspectedDependency`/`bottleneckType`) | 0.3 (略高, 允许多样性) | `rca.llm.lightweight-model` | 优先延迟与成本, 单选题够用即可 |

**分档的价值**: 6~8 次调用里, 只有 2~3 次 (推理 + Critic 校验) 真需要强推理; 其余是**"从有限枚举里选一个"**式的分类任务, 用轻量模型足够、且能把成本压下 3~5 倍.

### 2.3 具体模型选型建议

结合本项目实测 (真实推理模型偏慢, 已在 §1.3 上调超时) 与团队生态 (LiteLLM 网关), 给出如下**倾向性建议** (最终以线上灰度评测为准):

> **可选模型范围**: 本项目 LiteLLM 网关当前仅接入四个系列——**Claude 系列 (Anthropic)、Gemini 系列 (Google)、GLM-5 (智谱)、通义千问 (Qwen, 阿里)**. 下述选型仅在这四个系列内取舍.

| 档位 | 优先选型 | 备选 | 关键理由 |
|---|---|---|---|
| **REASONING** | GLM-5 / Claude 3.5 Sonnet | Gemini 1.5/2.0 Pro、Qwen-Max | ① 严格 JSON 结构化输出稳定; ② 具备真正的跨证据因果推理 (非纯模式匹配); ③ 中文告警场景表达自然. GLM-5 是当前应用侧默认, 已在 `application.yml` 落地; Claude 3.5 Sonnet 推理与结构化输出综合最优, 作为质量对标基准. |
| **LIGHTWEIGHT** | GLM-5-flash / Qwen-Turbo | Gemini 1.5/2.0 Flash、Claude 3.5 Haiku | ① 分类任务准确率与强模型差距 < 5%; ② 延迟 < 3s, 单次调用便宜 5~10 倍; ③ 结构化输出成熟. 优先用 GLM-5-flash / Qwen-Turbo 这类国产轻量档, 中文场景与成本更优. |
| **谨慎用** | 各系列的超小规格 (如 Qwen-Turbo 更低档) 承担 REASONING | — | 推理档不要图便宜下沉到轻量规格, 跨证据因果推理弱、易联想幻觉, 结构化输出解析失败率上升. |

**四个可选系列横向对比** (用于本项目选型权衡):

| 系列 | 推理质量 | 结构化输出稳定性 | 中文告警场景 | 延迟 | 成本 | 本项目定位 |
|---|---|---|---|---|---|---|
| **Claude 系列** | 极强 (跨证据因果推理最佳) | 极稳 | 良好 | 中 | 较高 | REASONING 首选/质量对标; Judge 独立评委 |
| **GLM-5 (智谱)** | 强 | 稳 | 优 (国产, 术语贴合) | 中 | 中 | REASONING 默认 (已落地 `application.yml`); flash 档做 LIGHTWEIGHT |
| **通义千问 (Qwen)** | 强 (Max 档) | 稳 | 优 | 低~中 | 低 | LIGHTWEIGHT 首选 (Turbo 档); Max 可做 REASONING 备选 |
| **Gemini 系列** | 强 | 稳 (需注意 JSON mode) | 良好 | 低 (Flash 快) | 中低 | REASONING/LIGHTWEIGHT 均可作备选, 灰度对比用 |

**特别说明**: 
- **不做微调**: 项目属于"证据驱动 + 结构化推理"场景, 提示词工程 + 分档路由 + 反幻觉体系已经充分, 不需要微调; 微调反而会带来"训练数据泄露"与"回归难以复现"的运维负担;
- **不做多模型投票 (self-consistency=3)**: 当前 `selfConsistencySamples` 字段留了但**默认不启用于线上**——3 倍成本换来的一致性提升在 Critic 硬校验 + 迭代取证前提下边际收益有限, 仅在评测阶段启用做稳定性度量;
- **温度**: 推理档 0.2、轻量档 0.3 是刻意选低——归因场景不需要"创意", 需要**可复现**.

### 2.4 通过 LiteLLM 网关接入的价值

选择所有模型统一走团队 LiteLLM 网关而非直连厂商 API, 是本项目的**架构级选型**:

- **模型可切换零成本**: 换模型只需改 `rca.llm.reasoning-model` 一行配置, 应用层零改动;
- **统一鉴权/限流/计费**: 不需要每个应用管理各家 API key;
- **OpenAI 兼容协议**: Spring AI OpenAI Starter 直接可用, 避免每家 SDK 各写一遍;
- **网关侧灰度**: 支持按流量比例把请求分流到不同模型, 是评测灰度的实现基础.

## 3. 多 Agent 架构下的通信与记忆机制

### 3.1 本项目当前的 Agent 协作模式

本项目**是多 Agent 系统, 但采用"编排式 (Orchestrated) 而非自主对话式 (Conversational)"**的协作模式. 具体来说:

- **6 个 Agent**: `LogAnalysisAgent`, `TraceMetricAgent`, `CodeContextAgent`, `RootCauseReasoningAgent`, `CriticAgent`, `ReportGeneratorAgent`;
- **通信媒介**: Java 强类型 Record (`LogAnalysisResult`, `TraceMetricResult`, `CodeContextResult`, `EvidenceBundle`, `ReasoningResult`, `CriticResult`, `RcaReport`);
- **协调者**: `Orchestrator` 独家控场, Agent 之间**不直接对话**, 全部经由 Orchestrator 转发数据.

这种设计是**刻意的**, 见架构文档 §2.2:

> **多 Agent 分工、单点推理**: 证据收集类 Agent 各司其职、并行执行; 只有 `RootCauseReasoningAgent` 做跨证据融合推理, 避免"多个 Agent 各自臆测、互相污染".

### 3.2 为什么本项目不用"自主对话式 (Agent-to-Agent Chat)"

对话式多 Agent (如 AutoGen、CrewAI 的对话链) 允许 Agent 之间用自然语言互相委派任务, 但在**故障排查**这类**高确定性 + 时效敏感**场景, 有三个致命问题:

1. **不可控的循环风险**: Agent A 让 B 补充证据, B 又反问 A, 极易打转;
2. **上下文爆炸**: 自然语言对话历史无边界增长, Token 成本失控;
3. **难以测试**: 每次对话轨迹都不同, 回归测试无从下手.

本项目用**强类型编排 + 规则驱动的迭代取证** (`InvestigationService`, 详细设计 §2.4) 替代自主对话, 既保留了"顺着线索深挖"的动态性, 又保证了确定性与可测性. 这是**在正确的场景选正确的抽象**.

### 3.3 假设场景: 若引入更复杂的多 Agent 场景, 如何设计通信与记忆

以下为**通用架构方案**, 供后续演进 (如引入历史案例 RAG Agent、拓扑分析 Agent、修复方案生成 Agent 并需要它们互相协作) 时参考.

#### 3.3.1 Agent 通信机制选型矩阵

| 模式 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| **A. 中心编排 (本项目当前)** | 流程固定、路径可枚举 | 强类型、可测、可容错 | 灵活性低, 新流程需改 Orchestrator |
| **B. 消息总线 (Event Bus)** | Agent 数量多、流程分支多、需异步解耦 | 松耦合, 易扩展新 Agent | 缺乏全局视图, 调试困难 |
| **C. 共享黑板 (Blackboard)** | Agent 需持续观察彼此产出、边看边改 | 全局共享上下文自然 | 并发写冲突, 竞态复杂 |
| **D. 对话协议 (Chat Protocol)** | 开放式探索、无预设流程 | 灵活, 涌现能力强 | 循环/幻觉/成本失控风险 |

**推荐组合演进路径** (假设本项目未来引入 RAG + 拓扑 + 修复建议 Agent):

```
第一阶段: A (中心编排) 单独
第二阶段: A + B (Orchestrator 主导, 部分辅助 Agent 走事件总线异步补充证据)
第三阶段: A + B + C (引入共享黑板作为 EvidenceBundle 的可增量扩展视图)
不推荐: D (对话协议) —— 除非确实需要开放式推理探索
```

#### 3.3.2 消息协议设计要点

若引入 B (Event Bus), 建议协议如下:

```java
// 事件基类, 强类型
public sealed interface AgentEvent permits EvidenceReadyEvent, HypothesisProposedEvent, ... {
    String correlationId();      // 全链路追踪 ID, 通常等于 caseId
    String producerAgent();       // 生产者 Agent 名
    long timestamp();
    int hopCount();               // 已经过的跳数, 用于死循环防护
}

public record EvidenceReadyEvent(
    String correlationId, String producerAgent, long timestamp, int hopCount,
    Evidence evidence, ConfidenceLevel producerConfidence
) implements AgentEvent {}
```

**关键约束**:
- 每个事件带 `hopCount`, 超过阈值 (如 10) 由总线直接丢弃并告警, 防止死循环 (对应 §1.4 死循环防护思路);
- 所有事件走**结构化**, 严禁自然语言载荷 (避免 D 模式的失控);
- 事件订阅显式声明 (静态注册), 不允许动态 wildcard, 方便测试与追踪.

#### 3.3.3 记忆机制 (Memory) 设计

多 Agent 系统的记忆通常分四层, 每层用途和存储介质不同:

| 层 | 名称 | 生命周期 | 存储 | 本项目对应 |
|---|---|---|---|---|
| **L1** | 短期工作记忆 (Working Memory) | 单次分析内 | JVM 堆 (Java Record) | `EvidenceBundle` 在 `Orchestrator` 内传递 |
| **L2** | 会话记忆 (Session Memory) | 单个案卷生命周期 | Redis (`RcaCacheService`) | Agent 中间结果按 `(caseId, agentName)` 缓存, TTL 30min, 保证重跑幂等 |
| **L3** | 长期记忆 (Long-term Memory) | 跨案卷持久化 | MySQL (`alert_case` 表) | 案卷落库, 未来可作为历史相似案例检索 (RAG) 的语料 |
| **L4** | 知识记忆 (Knowledge Memory) | 版本化管理 | 向量库 / 配置中心 | **本项目暂未使用**; 未来可用于 RAG 历史根因、SRE 排查手册向量化 |

**关于对话历史 / LangChain Memory 的取舍**:

- LangChain/Spring AI Memory 类抽象 (`ChatMemory`) 用于**多轮对话式** Agent, 本项目**不使用**;
- 原因: 本项目每次 LLM 调用是**独立 one-shot 请求** (`ChatClient.prompt().options(...).system(...).user(...).call()`), 不需要跨轮次上下文, 反而要**严格控制 Token 预算** (`ContextBudgetManager`), 保留历史反而会挤占证据空间;
- 若未来引入 L4 知识记忆, 应通过**独立 RAG Agent** 检索并把结果作为一条 `Evidence` 注入 `EvidenceBundle`, 而不是把它塞进 ChatMemory 里被动积累.

#### 3.3.4 一致性与并发控制

若 Agent 通过 B (总线) 或 C (黑板) 并发修改共享状态, 需要:

- **不可变数据结构**: `EvidenceBundle` 现已是 `record` 实现不可变, Agent 只产出**新的**副本, 不原地修改 (本项目已落地);
- **单一写者原则**: 每个字段只有一个 Agent 有权写 (如 `LogAnalysisAgent` 独占 `logResult`), 避免并发冲突;
- **版本号乐观锁** (若用共享黑板): 每次写带版本号, 版本冲突时由编排者决定合并策略.

#### 3.3.5 可观测性 —— 多 Agent 场景必备

单 Agent 出错可以看堆栈, 多 Agent 系统必须有**全链路视图**:

- **traceId 贯穿**: 从 `alertId` 生成的 traceId 挂在 MDC, 所有 Agent 日志自动携带;
- **事件序列可视化**: 若用总线, 事件流入 Kafka + Grafana Tempo, 可视化每次分析的完整 Agent 交互轨迹;
- **每 Agent 独立指标**: 复用本项目 `rca_agent_degraded_total{agent=xxx}` 的模式, 按 agent 维度切分所有指标.

### 3.4 本项目演进建议

短期 (0.5 年内): **保持当前中心编排模式**, 用数据驱动模型/提示词/阈值迭代 (评测方案见《06-Agent 工程测评方案》).

中期 (1 年内): 若引入**历史案例 RAG Agent**, 用 §3.3.3 的 L4 记忆方案, 通过独立 Agent + Evidence 注入的方式接入, 不改动主编排链路.

长期: 若确有需要 (如引入自愈变更 Agent), 再考虑 §3.3.1 的 A+B 混合模式, 用事件总线解耦可选辅助 Agent, 但主链路始终保持强类型编排.
