# AI 告警自动归因系统 —— 在线自我完善与经验沉淀方案

状态: 设计评审中
文档定位: 回答一个核心问题——**系统上线后, 如何借助人工反馈不断自我完善、提升归因诊断能力、沉淀诊断经验**. 本文档基于用户提出的"正/负反馈驱动 Agent 自我完善"思路, 结合本项目 (`ai-alert-rca-platform`) 的实际架构, 给出**分层、可落地、可控**的完整方案.

> 前置阅读: 《06-Agent 工程测评方案》(离线评测) 与本文档 (在线自我完善) 是一体两面——测评保证"不劣化", 自我完善保证"持续变好".

---

## 1. 核心理念: 构建"经验飞轮 (Experience Flywheel)"

用户提出的思路非常正确, 其本质是给归因系统装上一个**闭环学习飞轮**:

```
        ┌──────────────────────────────────────────────────────┐
        │                                                        │
        ▼                                                        │
  [归因报告产出] ──推送──> [SRE/开发人工确认] ──反馈──> [反馈结构化入库]
        ▲                                                        │
        │                                                        ▼
  [下次归因质量提升] <──经验注入── [经验沉淀与提炼] <──聚合分析──┘
```

飞轮转起来的关键: **每一次人工反馈都不能白费**, 都要沉淀为可被下次归因复用的"经验资产".

### 1.1 关键设计原则 (为什么不能简单地"让 Agent 自己学")

在生产环境做自我完善, 必须守住三条红线, 否则飞轮会**越转越坏**:

1. **不做在线自动微调/自主改 Prompt**: LLM 在线自改提示词或自动微调, 会导致行为**不可复现、不可回滚、难以测试**——一次坏反馈就可能污染整个系统. 本项目坚持 **"经验以数据资产形式沉淀 + 人工审核后灰度生效"**, 而非"模型自己偷偷变了".
2. **经验必须先过测评门禁再上线**: 任何"经验"(新规则、新提示词片段、新案例) 生效前, 必须先在《06 测评方案》的黄金集/回归集上验证不劣化 (对齐飞轮与测评的衔接).
3. **反馈质量本身要被治理**: 人工反馈可能有误、有噪声、有主观偏差. 需要对反馈做**可信度加权**, 避免个别错误反馈把系统带偏.

### 1.2 与用户原始思路的映射

| 用户思路 | 本方案落地机制 |
|---|---|
| "开发通过归因文档人工确认诊断正确, 给正面反馈" | §2 反馈采集: 报告尾部"采纳"按钮 → `FeedbackType.CONFIRMED` |
| "诊断有误, 给出建议" | §2 反馈采集: "驳回"按钮 + 结构化纠错 (真实根因/正确证据) |
| "只诊断出部分原因" | §2 反馈采集: "部分正确"按钮 + 补充遗漏方向 |
| "Agent 自我完善 skill" | §4 经验沉淀: 提炼为**规则/提示词片段/知识案例**三类"经验资产", 而非黑盒自改 |
| "沉淀诊断经验" | §4 经验库 (`rca_experience`) + §5 经验注入 (RAG 到 Reasoning Agent) |

## 2. 第一环: 人工反馈采集 (Feedback Collection)

### 2.1 反馈载体 —— 复用现有案卷状态机

本项目 `CaseService` 已内置反馈相关的终态流转 (无需新造状态机):

```
ANALYZED / ANALYZE_FAILED
        ├──> ACKNOWLEDGED   (已确认)
        ├──> RESOLVED       (已解决)
        └──> FALSE_POSITIVE (误报)
```

在此基础上, 通过 `CaseService.transitionTo` 已开放的通用流转能力接入反馈, **对现有状态机零改动**.

### 2.2 反馈的三种类型 (对应用户的三种场景)

在企微推送的报告 markdown 尾部提供交互按钮 (由 `NotificationGateway` 渲染), 回调到新增的反馈接口:

| 反馈类型 | 触发按钮 | 结构化内容 | 案卷状态流转 |
|---|---|---|---|
| **CONFIRMED (完全正确)** | ✅ 采纳 | 无需补充 | → `RESOLVED` |
| **PARTIAL (部分正确)** | ◐ 部分正确 | 遗漏的根因方向 + 补充说明 | → `ACKNOWLEDGED` |
| **REJECTED (诊断有误)** | ✗ 驳回 | 真实根因方向 + 正确证据线索 + 错误类型 | → `ACKNOWLEDGED` |
| **FALSE_POSITIVE (误报)** | 🔕 误报 | 无需分析的原因 | → `FALSE_POSITIVE` |

### 2.3 反馈数据结构

新增反馈实体 `AlertFeedbackEntity` (MySQL 表 `alert_feedback`):

```java
@Entity
@Table(name = "alert_feedback")
public class AlertFeedbackEntity {
    private Long id;
    private Long caseId;                    // 关联案卷
    private String fingerprint;             // 告警指纹, 用于同类告警经验复用
    private FeedbackType feedbackType;      // CONFIRMED / PARTIAL / REJECTED / FALSE_POSITIVE

    // === 系统当时的诊断快照 (用于对比学习) ===
    private String aiRootCauseSummary;      // AI 给出的根因
    private SuspectedCause aiRootCauseDirection;
    private ConfidenceLevel aiConfidence;

    // === 人工纠错内容 (REJECTED/PARTIAL 时填写) ===
    private SuspectedCause actualRootCauseDirection;  // 真实根因方向
    private String actualRootCauseSummary;            // 真实根因描述
    private String correctEvidenceHint;               // 正确的证据线索 (如"应看 GC 指标")
    private String errorCategory;                     // 错误类型 (见 §2.4)
    private String suggestion;                        // 开发的自由建议

    // === 反馈治理字段 ===
    private String feedbackBy;              // 反馈人 (用于可信度加权)
    private Integer feedbackerLevel;        // 反馈人级别 (资深 SRE 权重更高)
    private LocalDateTime feedbackAt;
    private Boolean adoptedToExperience;    // 是否已提炼为经验资产
}
```

### 2.4 错误类型分类 (Error Taxonomy)

驳回时强制选择错误类型, 便于后续归因分析定位是**哪个能力维度**出了问题 (对齐《06》能力矩阵):

| 错误类型 | 含义 | 指向的能力维度 | 典型改进动作 |
|---|---|---|---|
| `WRONG_DIRECTION` | 根因方向完全错 | C5 推理 / C2 取证转向 | 补充迭代取证决策树规则 |
| `INCOMPLETE` | 只诊断出部分 | C5 推理 / C1 路由 | 检查是否漏激活 Agent |
| `WRONG_EVIDENCE` | 引用证据不当 | C5 推理 | 强化 Critic 硬规则 / Prompt |
| `MISSED_ROUTING` | Agent 激活集不对 | C1 意图理解 | 补充路由规则 |
| `OVER_CONFIDENT` | 置信度虚高 | C5 自校验 | 调整置信度校准 |
| `HALLUCINATION` | 编造了证据/结论 | C5 反幻觉 | 升级 Critic / 红队集补样本 |

### 2.5 降低反馈门槛 (让开发愿意反馈)

飞轮能否转起来, 取决于**反馈率**. 为此:

- **一键正反馈**: "采纳"零成本, 一键完成 (大多数正确案例只需这一步);
- **纠错渐进式**: 驳回时先只选"错误类型" (必填), 真实根因/建议为选填, 降低负担;
- **反馈激励**: 反馈纳入 SRE 值班考核正向指标; 被提炼为经验的反馈给贡献者署名;
- **就地反馈**: 直接在企微卡片交互, 不跳转其它系统.

## 3. 第二环: 反馈治理与聚合分析 (Feedback Governance)

**原始反馈不能直接变成经验**, 必须先经过治理, 否则噪声/错误反馈会污染系统.

### 3.1 反馈可信度加权

每条反馈计算一个可信度分 `credibility`:

```
credibility = w_level × 反馈人级别权重     // 资深 SRE = 1.0, 一般开发 = 0.6
            + w_consensus × 同类共识度      // 同指纹下多人反馈一致则加权
            + w_verified × 是否有事后验证    // 若后续故障复盘验证了该反馈, 权重最高
```

- 单条低可信度反馈**不足以**驱动经验沉淀;
- 同一指纹下**多条一致的高可信反馈**才触发经验提炼 (对抗个别误判).

### 3.2 反馈聚合视图

按维度聚合, 定期 (如每日/每周离线任务) 产出分析:

| 聚合维度 | 分析产出 | 驱动的改进 |
|---|---|---|
| 按 **错误类型** | 各类错误占比趋势 | 定位系统最薄弱的能力维度 |
| 按 **告警指纹** | 哪些同类告警反复被驳回 | 高频 badcase 优先沉淀经验 |
| 按 **服务** | 哪些服务归因准确率低 | 该服务可能需专属规则/知识 |
| 按 **根因方向** | 哪个 `SuspectedCause` 最常误判 | 校准 `HypothesisVerifier` 阈值 |
| 按 **置信度校准** | HIGH 结论实际正确率 | 调整置信度分级策略 |

### 3.3 与测评体系的对接

聚合出的**高价值 badcase** 自动流入《06 测评方案》的**回归集**:

```
[REJECTED 反馈 + 高可信度] --自动脱敏--> [生成标注样本] --PR--> regression-set
```

这样每一个真实的诊断错误, 都会变成一条**永久的回归测试用例**, 保证同样的错误不再犯第二次.

## 4. 第三环: 经验沉淀 (Experience Distillation)

这是用户所说 "Agent 自我完善 skill、沉淀诊断经验" 的核心落地. 关键设计: **经验以三类结构化资产沉淀, 而非让模型黑盒自改**.

### 4.1 三类经验资产

| 经验类型 | 形态 | 生效方式 | 举例 |
|---|---|---|---|
| **A. 规则型经验** | 确定性规则 (JSON/配置) | 注入 `RoutingEngine` / `InvestigationService` 决策树 / `HypothesisVerifier` 阈值 | "服务 X 的 Redis 超时告警, 优先查连接池而非网络" |
| **B. 提示词型经验** | Prompt 片段 (few-shot 示例) | 注入 `RootCauseReasoningAgent` 的 system/user prompt | "曾有案例: 日志显示 Redis 超时但实为 GC 停顿拖累, 需交叉验证 GC 指标" |
| **C. 案例型经验** | 结构化历史案例 (向量化) | RAG 检索, 作为一条 `Evidence` 注入 `EvidenceBundle` | 完整的"告警特征 → 真实根因 → 排查路径"案例 |

**为什么分三类**: 不同经验适合不同载体——
- 能明确成 if-then 的 → **规则** (确定性最强, 零 LLM 成本, 最可控);
- 需要"启发式提醒"的 → **提示词** (影响 LLM 判断倾向, 但不写死);
- 完整排查经验 → **案例** (供相似场景类比, 最灵活).

### 4.2 经验库数据结构

新增 `ExperienceEntity` (MySQL 表 `rca_experience` + 向量库):

```java
public class ExperienceEntity {
    private Long id;
    private ExperienceType type;            // RULE / PROMPT / CASE
    private String scope;                    // 适用范围: 全局 / 服务名 / 告警类型
    private String matchPattern;             // 匹配条件 (指纹特征/服务/告警类型)

    // 经验内容 (按 type 解释)
    private String content;                  // RULE: JSON规则; PROMPT: few-shot文本; CASE: 结构化案例
    private float[] embedding;               // CASE 型的向量表示, 供相似检索

    // 溯源与治理
    private Long sourceFeedbackId;           // 来源反馈
    private Double credibility;              // 可信度 (继承自反馈聚合)
    private ExperienceStatus status;         // DRAFT / VALIDATING / ACTIVE / DEPRECATED
    private Double evalUplift;               // 上线前评测的质量提升值
    private LocalDateTime createdAt;
    private LocalDateTime activatedAt;
}
```

### 4.3 经验提炼流程 (人工审核在环)

```
[高可信 badcase 聚合]
        │
        ▼
[经验候选生成] ──可用 LLM 辅助提炼 (把反馈转成规则/提示词草稿)──> status=DRAFT
        │
        ▼
[SRE 专家审核] ──人工判断经验是否正确、可泛化──> 通过则 status=VALIDATING
        │
        ▼
[离线评测验证] ──在黄金集+回归集上跑, 对比启用前后──> evalUplift ≥ 0 则 status=ACTIVE
        │
        ▼
[灰度生效] ──先 10% 流量, 观察后全量──> 若线上指标恶化则 status=DEPRECATED
```

**关键**: LLM 只用于"把反馈草拟成经验候选" (辅助), **最终是否沉淀由 SRE 审核 + 评测门禁决定**, 系统不会自主把未经验证的东西写进决策链.

## 5. 第四环: 经验注入 (Experience Injection) —— 让下次归因更强

经验沉淀后, 必须能被下次归因**自动复用**, 飞轮才闭环. 三类经验分别注入不同环节:

### 5.1 规则型经验注入

- **注入点**: `RoutingEngine.route` (激活 Agent 决策)、`InvestigationService.nextTarget` (取证转向决策树)、`HypothesisVerifier` 阈值;
- **方式**: 规则从经验库加载 (可放七彩石热更新), 匹配当前告警的 `scope`/`matchPattern` 时生效;
- **示例**: 经验"服务 X 的 Redis 超时优先查连接池" → 当告警服务=X 且疑似 REDIS 时, 迭代取证决策树优先插入"连接池指标"验证.

### 5.2 提示词型经验注入

- **注入点**: `RootCauseReasoningAgent.buildUserPrompt` / system prompt;
- **方式**: 检索与当前告警匹配的高可信 PROMPT 经验, 作为 **few-shot 提醒**拼接到 prompt (受 `ContextBudgetManager` Token 预算约束, 只取 Top-K 最相关);
- **示例**: 在 prompt 中追加 "历史相似案例提醒: 日志出现 Redis 超时字样时, 曾有 3 例实为 GC 停顿导致, 请务必交叉验证 GC 指标后再定论".

### 5.3 案例型经验注入 (RAG)

这正是《05 关键议题补充》§3.4 提到的 **"历史案例 RAG Agent"** 的落地:

- **注入点**: 作为一个**新增的可选证据收集 Agent** (`HistoricalCaseAgent`), 在 fan-out 阶段并行执行;
- **方式**: 用当前告警特征 (服务/告警类型/日志指纹/疑似方向) 生成 query embedding, 从经验库向量检索 Top-K 相似历史案例, 封装为 `Evidence` (type=`HISTORICAL_CASE`) 注入 `EvidenceBundle`;
- **防幻觉一致性**: 历史案例作为**普通证据**参与推理, 同样受 Critic 引用校验约束, 不享受特权;
- **架构友好**: 完全复用现有 fan-out + Evidence 机制, **不改动主编排链路** (对齐 05 文档"通过独立 Agent + Evidence 注入"的演进建议).

### 5.4 注入的安全约束

- **经验带可信度**: 低可信经验只作弱提示, 不能直接改变确定性判定;
- **经验可追溯**: 报告中若采信了历史经验, 在 `Evidence.sourceRef` 标注来源经验 ID, 保持全链路可追溯;
- **经验可淘汰**: 若某经验注入后线上准确率不升反降, 自动 `DEPRECATED`, 防止过时经验长期误导.

## 6. 完整闭环与关键指标

### 6.1 端到端飞轮流程图

```
 告警 ──> [归因分析] ──(注入历史经验)──┐
                │                      │
                ▼                      │ 经验注入(§5)
          [归因报告推送]                │
                │                      │
                ▼                      │
     [SRE 反馈: 确认/部分/驳回] (§2)     │
                │                      │
                ▼                      │
       [反馈治理与聚合] (§3)            │
          │            │               │
          ▼            ▼               │
   [回归集沉淀]   [经验提炼] (§4)        │
   (对接《06》)    │                    │
                  ▼                    │
          [SRE审核+评测门禁]            │
                  │                    │
                  ▼                    │
            [经验库 ACTIVE] ───────────┘
```

### 6.2 飞轮健康度指标

飞轮转得好不好, 用以下指标衡量:

| 指标 | 含义 | 目标趋势 |
|---|---|---|
| **反馈率** | 有人工反馈的案卷 / 总案卷 | ≥ 40% (飞轮的燃料) |
| **正反馈率 (采纳率)** | CONFIRMED / 有反馈案卷 | 持续上升 (核心北极星) |
| **经验转化率** | 沉淀为 ACTIVE 经验的反馈 / REJECTED 反馈 | 稳定 (说明错误在被吸收) |
| **经验命中率** | 归因时命中历史经验的案卷比例 | 随经验库增长上升 |
| **经验有效性** | 命中经验的案卷 vs 未命中的准确率差 | > 0 (证明经验真的有用) |
| **同类错误复发率** | 已沉淀经验的错误类型再次发生率 | 持续下降 (飞轮生效的铁证) |

### 6.3 与测评方案的分工

| | 《06 测评方案》 | 本方案 (07 自我完善) |
|---|---|---|
| 目标 | 保证**不劣化** (回归防护) | 保证**持续变好** (经验飞轮) |
| 触发 | 每次代码/模型/Prompt 改动 | 每次线上人工反馈 |
| 数据 | 黄金集 + 回归集 (静态) | 线上反馈流 (动态) |
| 关系 | 经验上线前的**质量门禁** | 回归集的**数据来源之一** |

二者构成"**评测守底线, 飞轮拉上限**"的完整质量体系.

## 7. 业界最新相似方案参考与借鉴 (2024-2025)

用户提出的"反馈驱动自我完善"思路, 与业界近两年的研究/产品高度一致. 本节梳理**学术界的记忆与自我改进范式**、**工业界的 RCA 产品实践**, 并说明**哪些可以直接借鉴到本项目**、哪些需要审慎.

### 7.1 学术界: Agent 自我改进的四条技术路线

近两年 LLM Agent 自我改进的研究可归纳为四条路线, 按"可控性从高到低"排列:

| 路线 | 代表工作 | 核心机制 | 对本项目的适用性 |
|---|---|---|---|
| **① 经验记忆 (Experiential Memory)** | **ExpeL** (Experiential Learning, AAAI 2024)、**Generative Agents** (记忆流+反思, Stanford) | 从历史成败中提炼**自然语言经验/洞察**, 检索后注入 prompt, **不改模型权重** | ★★★ 最契合. 本项目 §4 "经验资产" 就是这一思路的工程化 |
| **② 反思 (Reflection / Reflexion)** | **Reflexion** (NeurIPS 2023)、Self-Refine | 任务失败后生成**语言化反思**存入情景记忆, 下次任务前读取以规避重复错误 | ★★☆ 可借鉴. 本项目的反思由**人工反馈提供** (更可信), 而非 Agent 自我反思 (见 §7.4) |
| **③ 技能库 (Skill Library)** | **Voyager** (Minecraft, NVIDIA)、A-MEM | 把成功的解法沉淀为**可复用技能/记忆条目**, 动态组合, 技能库随经验增长 | ★★★ 高度契合用户"完善 skill"的原话. 对应本项目 §4 的"规则型经验" |
| **④ 在线强化/微调 (RL / Fine-tune)** | **TN-AutoRCA** (arXiv 2025, 电信网络 RCA + RL)、RISE、自博弈 | 用反馈作为奖励信号在线 RL 或周期性微调, **改变模型权重** | ★☆☆ 谨慎. 生产不可复现/难回滚, 本项目**明确不采用** (见 §1.1、§8) |

### 7.2 记忆架构: 从 MemGPT 到 A-MEM 的可借鉴点

Agent 记忆领域 2024-2025 的演进 (Generative Agents → Reflexion → Voyager → MemGPT/Letta → A-MEM) 给出几个可直接落地的设计:

- **记忆流 + 反思 + 规划三层 (Generative Agents)**: 与本项目 L1~L4 记忆分层 (见《05》§3.3.3) 思路一致; 其"定期反思把琐碎记忆提炼成高层洞察"正是本项目 §4 "经验提炼" 的理论依据;
- **A-MEM 的动态记忆结构 (2025)**: 不用预设固定 schema, 而是让记忆条目之间**自动建立关联链接** (类似 Zettelkasten)——可借鉴到本项目**经验库的组织**: 相似案例/规则之间建立引用关系, 检索时可顺链扩展;
- **记忆的"检索-遗忘"机制**: MemGPT 分级存储 + A-MEM 记忆演化提示——对应本项目**经验淘汰** (§5.4 过时经验 `DEPRECATED`), 避免记忆库无限膨胀且被过时经验误导;
- **重要提醒**: 学术记忆系统多为**单 Agent 自主积累**, 缺乏质量把关; 本项目在其上加了**人工审核 + 评测门禁 + 可信度加权** (§3、§4.3), 是面向生产的强化.

### 7.3 工业界: RCA 产品/论文的实践借鉴

| 来源 | 关键实践 | 本项目可借鉴点 |
|---|---|---|
| **Microsoft RCACopilot** (ASE 2024, 已在 30+ 团队部署) | ① 用历史事件构建**诊断知识库**, 新事件检索相似历史案例; ② on-call 工程师的处置作为反馈回流; ③ 分阶段: 收集诊断信息 → 预测根因类别 | **强验证本项目 §5.3 案例 RAG 路线**——微软大规模生产验证了"历史相似案例检索"对 RCA 的价值; 其"诊断信息收集与推理分离"也印证本项目"证据收集 Agent 与推理 Agent 分工" |

### 7.4 本项目相对业界方案的差异化与增强

综合业界方案, 本项目在借鉴的同时做了三点**面向生产的增强**, 这也是给评审的核心论证:

1. **反馈来源: 人工可信反馈 > Agent 自我反思**
   Reflexion 让 Agent 自己反思自己的错误, 但在 RCA 场景中——**如果 Agent 能自己发现根因错了, 它一开始就不会错**. 故本项目的"反思信号"来自 **SRE 人工反馈** (§2), 比 Agent 自省更可信; Agent 自省仅保留在 `CriticAgent` 的**当次幻觉拦截**上 (已实现), 不用于跨案卷学习.

2. **学习方式: 数据资产沉淀 > 在线权重更新**
   业界 RL/微调路线 (TN-AutoRCA 等) 效果好但生产风险高. 本项目选择 **ExpeL/Voyager 式的"经验外置"** 路线——经验存在库里 (规则/提示词/案例), 模型权重不变, 保证**可复现、可回滚、可测试、可审计**, 这是生产系统的刚需.

3. **质量把关: 人工审核 + 评测门禁 > 自主生效**
   学术记忆系统的经验自动积累自动生效, 生产中会**放大坏经验**. 本项目在经验生效链路上加了 **SRE 审核 (§4.3) + 离线评测验证 (对接《06》) + 灰度 (§5) + 可信度加权 (§3.1) + 过时淘汰 (§5.4)** 五道关卡.

### 7.5 可选增强项 (来自业界, 供后续评估)

以下是业界方案中**本项目当前未纳入、但值得后续评估**的增强点:

- **经验条目自动关联 (A-MEM 式)**: 让经验库的规则/案例之间建立引用图, 检索一条可顺链带出相关经验, 提升召回;
- **反思 Agent 辅助提炼 (Reflexion 式)**: 在 §4.3 "经验候选生成" 环节, 用一个专门的 Reflexion 风格 LLM 把"AI 诊断快照 + 人工纠错"对比反思成结构化经验草稿, 提升提炼效率 (仍需人工审核);
- **降噪规则自学习 (Mezmo 式)**: 把 `FALSE_POSITIVE` 反馈回流到告警治理层 `SuppressionEngine`, 自动优化抑制规则, 减少无效归因;
- **根因分类器 (RCACopilot 式)**: 用积累的"告警特征→真实根因方向"标注数据, 训练一个轻量分类器做**根因方向预筛**, 为迭代取证提供更准的起点 (替代/补充当前仅靠日志 Agent 的 `suspectedDependency`).

## 8. 落地路线 (Roadmap)

分四期递进, 每期都能独立产生价值, 避免一次性大改:

| 期 | 目标 | 交付物 | 价值 |
|---|---|---|---|
| **P1: 反馈采集** | 把反馈收上来 | `alert_feedback` 表 + 企微反馈按钮 + 反馈接口 + 采纳率看板 | 首先获得"系统到底准不准"的真实数据 |
| **P2: 反馈→回归集** | 让错误不再复发 | 反馈治理聚合 + badcase 自动入回归集 (对接《06》) | 每个错误变成永久回归用例 |
| **P3: 案例型经验 (RAG)** | 让系统会"类比" | 经验库 + `HistoricalCaseAgent` + 向量检索注入 | 相似故障复用历史排查经验 |
| **P4: 规则/提示词经验** | 让系统会"进化" | 经验提炼流程 + 审核台 + 灰度生效 + 经验有效性度量 | 完整飞轮闭环, 诊断能力自增强 |

**建议**: 先做 P1+P2 (投入小、见效快、风险低), 用积累的反馈数据验证飞轮价值后, 再投入 P3+P4 的经验沉淀与注入.

## 9. 关键取舍总结 (给评审的话)

1. **为什么不让 Agent 在线自动微调/自改 Prompt?**
   生产环境不可复现、不可回滚、难测试是大忌. 本方案让经验**以数据资产沉淀 + 人工审核 + 评测门禁 + 灰度生效**, 既实现"自我完善"的效果, 又守住工程可控性.

2. **为什么经验分规则/提示词/案例三类?**
   不同经验适合不同载体, 能写死的用规则 (最可控), 启发式的用提示词, 完整经验用案例 RAG. 避免"一把梭"塞进 Prompt 导致 Token 爆炸和判断漂移.

3. **为什么强调反馈治理?**
   人工反馈有噪声、有主观、可能出错. 不做可信度加权和共识校验, 飞轮会被个别坏反馈带偏——"垃圾进, 垃圾出".

4. **和测评方案的关系?**
   本方案 (拉上限) 与《06》(守底线) 互为支撑: 反馈产出的 badcase 喂给测评回归集, 经验上线前必须过测评门禁. 二者缺一, 飞轮都不安全.

5. **和业界方案的关系?**
   本方案的经验飞轮与业界 ExpeL/Voyager (经验外置)、Reflexion (反思记忆)、RCACopilot (历史案例检索, 30+ 团队生产验证) 一脉相承 (见 §7); 差异在于把"Agent 自主学习"替换为"**人工可信反馈驱动 + 数据资产沉淀 + 评测门禁 + 灰度生效**", 用工程可控性换取生产安全, 而非追求学术上的全自动.
